# app-infrastructure

GitOps repository for infrastructure deployments. Managed by the infrastructure MCP server.

## Folder-Based Model

All infrastructure is deployed from folders on `main`. No environment branches.

```
{app_id}/{app_name}/{environment}/{pattern}/
  terraform.tfvars.json    # Generated by MCP server or promotion workflow
  backend.hcl              # Terraform state key
```

## Environments

| Environment | Purpose | How tfvars arrive |
|-------------|---------|-------------------|
| `prototype` | Developer sandbox | MCP `provision` tool pushes directly to main |
| `dev` | Development integration | `promote.yaml` PR or `push_tfvars` |
| `tst` | Testing/QA | `promote.yaml` PR or `push_tfvars` |
| `stg` | Staging/pre-prod | `promote.yaml` PR or `push_tfvars` |
| `prd` | Production | `promote.yaml` PR or `push_tfvars` |

## How It Works

1. Developer provisions infrastructure via Claude Code + MCP server (`provision` tool, prototype only)
2. MCP server pushes tfvars to `{app_id}/{app_name}/prototype/{pattern}/` on main
3. Push triggers `terraform-apply.yaml` which runs terraform plan/apply
4. DevOps promotes to higher environments via `promote.yaml` workflow (creates PR)
5. PR merge triggers `terraform-apply.yaml` for the target environment

## Promotion

PR-based promotion via the `promote.yaml` workflow. DevOps triggers from GitHub Actions UI.

```
prototype → dev → tst → stg → prd
```

The workflow:
1. Reads source tfvars from the source environment folder
2. Adjusts sizing based on the pattern YAML config (infers t-shirt size, applies target env sizing)
3. Generates new backend.hcl with target environment state key
4. Creates a branch and PR with the adjusted tfvars
5. PR merge triggers terraform-apply for the target environment

Not all apps move beyond prototype — it's a stable state.

## State Management

Terraform state is stored in Azure Storage with per-deployment isolation.

**Key format:** `{app_id}/{business_unit}/{environment}/{project}/{pattern}-{name}/terraform.tfstate`

Configured via `backend.hcl` in each pattern directory. The storage account, container, and resource group are injected by the workflow via `-backend-config` CLI flags.

## Pattern Repo Mapping

The workflow determines which Terraform repo to check out based on pattern category:

| Category | Pattern Repo | Terraform Directory |
|----------|-------------|---------------------|
| Single-resource | `terraform-azurerm-{pattern}` | `pattern/` |
| Composite | `terraform-pattern-{pattern}` | `.` (root) |

Current composite patterns: `web_backend`

## Workflows

### terraform-apply.yaml
- **Trigger:** Push to `main` when any `**/terraform.tfvars.json` changes
- **Concurrency:** One apply per `app_id + environment + pattern` combination
- **Credentials:** Per-environment Azure service principals via OIDC (`AZURE_CLIENT_ID_prototype`, `_dev`, `_tst`, `_stg`, `_prd`)
- **Outputs:** Terraform outputs are captured in the GitHub Actions step summary

### promote.yaml
- **Trigger:** `workflow_dispatch` (manual, DevOps team)
- **Inputs:** application_id, application_name, pattern, source_environment, target_environment
- **Validation:** Target must be higher than source in the promotion ordering
- **Output:** Creates a PR with adjusted tfvars for the target environment
